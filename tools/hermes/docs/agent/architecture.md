# Hermes Architecture and Introduction

Hermes lets you annotate safe and `unsafe` Rust code with inline annotations in Lean 4 for:
- Specifying function behavior (preconditions and postconditions)
- Defining field invariants
- Defining `unsafe` trait invariants
- Proving correctness:
  - Proving that function preconditions are satisfied (when calling a function)
  - Proving that function postconditions are satisfied (when writing a function)
  - Proving that type invariants are maintained (when writing a function)
  - Proving that `unsafe` trait invariants are maintained (when implementing the trait)

Hermes translates Rust code to Lean 4 using the Aeneas toolchain. It combines Lean 4 generated by Aeneas with annotations written inline in Rust source code and uses Lean to verify the correctness of proofs.

This document walks through a complete example, introducing Hermes concepts one by one. For the full, working example, see `examples/anatomy.rs`.

## 1. Type Invariants (`isValid`)

`isValid` defines a structural invariant for a type.

Under the hood, Hermes will automatically generate a `Hermes.IsValid` instance for this type mapping to the provided expression. You can apply it in proofs using `simp_all [Hermes.IsValid.isValid]`.

```rust
/// ```hermes
/// isValid self := self.val.val > 0
/// ```
pub struct PositiveUsize {
    val: usize,
}
```

## 2. Trait Invariants (`isSafe`)

`isSafe` defines an invariant for an `unsafe` trait. In this example, we enforce that `Self` has an alignment of exactly 1.

```rust
/// ```hermes
/// isSafe : ∀ {{_sz : Hermes.core.marker.Sized Self}} {{tl : Hermes.HasStaticLayout Self}},
///   tl.layout.align.val.val = 1
/// ```
pub unsafe trait Unaligned: Sized {}
```

## 3. Axioms (`unsafe(axiom)`)

A specification provided with `unsafe(axiom)` tells Aeneas to skip analyzing the body of the function. Hermes will accept the specification without proof.

```rust
/// ```hermes, unsafe(axiom)
/// requires: a.val + b.val <= Usize.max
/// ensures: ret.val = a.val + b.val
/// ```
pub unsafe fn fast_add(a: usize, b: usize) -> usize {
    // ...
}
```

In Hermes, anonymous `requires` and `ensures` bounds are grouped and compiled into a single field named `h_unnamed`. These are namespaced to pre-conditions and post-conditions independently.

## 4. Function Specifications (`requires` and `ensures`)

Now we define a function `get_unaligned_fast_pad` that binds our concepts together.

```rust
/// ```hermes
/// requires (h_is_safe): Unaligned.Safe T Inst
/// ensures (h_align_eq): ret.val.val = 1
```

Here we define a named pre-condition `h_is_safe` and a named post-condition `h_align_eq`. Notice the type `Unaligned.Safe T Inst`. When you define an `isSafe` block on a trait `Unaligned`, Hermes generates a `.Safe` predicate parameterized by the type `T` and the specific trait implementation `Inst`.

## 5. The `proof context` Block

`proof context` is a special block in Hermes. Any bindings defined here are made available to ALL subsequent `proof (...)` blocks for the function. It prevents duplicating common setup.

```rust
/// proof context:
///   -- Prove that `fast_add(x, 0)` is the identity function.
///   have fast_add_zero : ∀ (x : Aeneas.Std.Usize), Hermes.IsValid.isValid x →
///     fast_add x 0#usize = Result.ok x := by
///     intro x h_valid
///
///     -- Supply { ... } explicitly for the anonymous `requires` bound.
///     -- Hermes injects implicit preconditions `h_<arg>_is_valid` for all args.
///     have ho := fast_add.spec x 0#usize { h_a_is_valid := h_valid, h_unnamed := by scalar_tac }
```

**Cracking Monadic Executions:**
To prove that `fast_add x 0` strictly returns `Result.ok x` and never fails:
```rust
///     cases h_eq : fast_add x 0#usize <;> simp_all
///     rename_i r; have _ := ho.h_unnamed; scalar_tac
```
Because the constraint `ho` requires it to succeed, `cases ... <;> simp_all` instantly prunes the impossible error states.

### Typeclasses and Trait Dictionaries

```rust
///   -- Marker traits like `Sized` translate to empty typeclasses.
///   -- Instantiate them with the anonymous constructor `⟨⟩`.
///   have h_sized : Hermes.core.marker.Sized T := ⟨⟩
///   
///   -- Aliasing complex layouts to a named binding first.
///   -- (In practice, rely on implicit variables instead of building manually).
///   have h_layout : Hermes.HasStaticLayout T := {
///     layout := { size := 1#usize, align := ⟨1#usize, by decide, 0, by rfl⟩, sizeAligned := by decide }
///   }
///
///   -- External Rust calls like `align_of` are translated as opaque axioms.
///   -- We inject a `_spec` theorem tying it to `HasStaticLayout`.
///   have h_aspec : core.mem.align_of T = Result.ok _ :=
///     @core_mem_align_of_spec T h_sized h_layout
///
///   -- Using the trait dictionary `UnalignedInst` to unlock `isSafe` theorems.
///   have h_s := h_is_safe (Inst := UnalignedInst)
///   have h_safe_eq := h_s.isSafe (_sz := h_sized) (tl := h_layout)
```

### Implicit Identifiers (`ret` and `h_returns`)

In Orthogonal WP Proofs, Hermes automatically destructures the `Result` into variables like `ret` and `arg'`, and provides an `h_returns` hypothesis binding them:
`h_returns : get_unaligned_fast_pad ... = ok ret`

When unwrapping monadic chains, standard rewrites fail. Always apply `rw [..., Aeneas.Std.bind_tc_ok]`.

```rust
///   have h_fast_pad_result : ret.val.val = 1 := by
///     unfold get_unaligned_fast_pad at h_returns
///     rw [h_aspec, Aeneas.Std.bind_tc_ok] at h_returns
///     have h_valid : Hermes.IsValid.isValid h_layout.layout.align.val := by simp_all [Hermes.IsValid.isValid]
///     rw [fast_add_zero _ h_valid, Aeneas.Std.bind_tc_ok] at h_returns
///     cases ret; simp_all
```

## 6. Proving Correctness Goals

We can usually dispatch the implicitly demanded `h_ret_is_valid` structure and our named bounds cleanly using values built in our `proof context` or via `simp_all`.

```rust
/// proof (h_ret_is_valid):
///   simp_all [Hermes.IsValid.isValid]
///
/// proof (h_align_eq):
///   simp_all
```

## 7. The Progress Barrier

Why couldn't we just use the Lean `eval_progress` tactic or the standard `progress` macro inside `h_ret_is_valid`?

Hermes separates verification into progress and correctness. During correctness blocks (`proof (h_ret_...)`), the Weakest Precondition states are destructured. The goal state for `h_progress` is an existential equality (`⊢ ∃ y, m = ok y`), NOT a WP `spec`. 

To bridge the gap between `spec` constraints and the existential constraint `∃ y, ...`, we use `spec_imp_exists`.

```rust
/// proof (h_progress):
///   unfold get_unaligned_fast_pad
///
///   -- (Scaffolding elided ... )
///   have h_align : ∃ align, core.mem.align_of T = Result.ok align :=
///     ⟨_, @core_mem_align_of_spec T h_sized h_layout⟩
///   rcases h_align with ⟨align, h_align_eq⟩
///   
///   -- Bridge generated WP `spec` to existential `∃ y, ...`
///   have h_add : ∃ padded, fast_add align 0#usize = Result.ok padded := by
///     have ho := fast_add.spec align 0#usize {
///       h_a_is_valid := by simp_all [Hermes.IsValid.isValid]
///       h_b_is_valid := by simp_all [Hermes.IsValid.isValid]
///       h_unnamed := by scalar_tac
///     }
///     rcases Aeneas.Std.WP.spec_imp_exists ho with ⟨padded, h_eq, _⟩
///     exact ⟨padded, h_eq⟩
///   rcases h_add with ⟨padded, h_add_eq⟩
///   
///   -- Chain evaluated states seamlessly using `bind_tc_ok`:
///   rw [h_align_eq, Aeneas.Std.bind_tc_ok, h_add_eq, Aeneas.Std.bind_tc_ok]
///   exact ⟨_, rfl⟩
/// ```
/// pub unsafe fn get_unaligned_fast_pad<T: Unaligned>() -> PositiveUsize {
///     let align = core::mem::align_of::<T>();
///     let padded = fast_add(align, 0);
///     PositiveUsize { val: padded }
/// }
```
